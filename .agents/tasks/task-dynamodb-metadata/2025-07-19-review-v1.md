# DynamoDB metadata table and state machine I/O handling

This change adds a DynamoDB table to track audio processing pipeline executions and wires it into the Step Functions state machine as bookend states: a PutItem before Polly synthesis records the job as PROCESSING, and an UpdateItem afterward marks it COMPLETED. The table uses on-demand billing, AWS-managed encryption, point-in-time recovery, and a RETAIN deletion policy. Permissions are granted via CDK's `grantReadWriteData`, and 10 new CDK assertion tests validate the infrastructure.

**Watch for:** Over-broad IAM grant on the metadata table (confirmed) ‚Äî `grantReadWriteData` adds permissions well beyond what the two task states actually exercise. Missing error-handling path ‚Äî if Polly fails, the metadata record stays permanently in PROCESSING with no mechanism to update it (confirmed). The `audioId` partition key uses the raw S3 object key, which couples table access patterns to bucket structure and makes re-processing the same file a silent overwrite (likely).

## High-level view

The DynamoDB table is single-key, on-demand, encrypted, PITR-enabled ‚Äî no sort key, no GSIs, no TTL. The choice of `audioId` as a bare partition key without a sort key means the table can only store one record per S3 object key; re-uploads overwrite previous metadata silently.

The state machine chain is now WriteMetadata ‚Üí Synthesize Speech ‚Üí UpdateStatus ‚Üí Done, a linear happy path with no error handling. A Polly failure leaves the DynamoDB record stuck in PROCESSING indefinitely. The architecture doc describes this table as "a queryable audit trail of all pipeline executions and their outcomes" ‚Äî that promise depends on status accuracy, which this implementation doesn't guarantee on failure.

IAM permissions use `grantReadWriteData` which adds BatchGetItem, Query, Scan, DeleteItem, and other actions the state machine never uses. The task-level policies (PutItem, UpdateItem) generated by the CDK task constructs are already scoped correctly, making the broader grant redundant but additive.

## IAM permission surface

The synthesized CloudFormation shows three separate policy statements touching DynamoDB. The two task-generated statements (one for PutItem, one for UpdateItem) are correctly scoped to the table ARN via `Fn::Join`. The third statement comes from `grantReadWriteData` and adds a broad set of actions (BatchGetItem, Query, Scan, ConditionCheckItem, BatchWriteItem, PutItem, UpdateItem, DeleteItem, DescribeTable, plus GetRecords and GetShardIterator for streams) on `Fn::GetAtt [Table, Arn]`. The grant violates least-privilege and adds unnecessary attack surface if the role is ever assumed in a broader context.

```typescript
metadataTable.grantReadWriteData(stateMachine);
```

This single line produces ~20 additional IAM actions. Replacing it with `metadataTable.grant(stateMachine, 'dynamodb:PutItem', 'dynamodb:UpdateItem')` would match the actual access pattern. Alternatively, since the CDK task constructs already generate their own scoped policies, the explicit grant could be removed entirely ‚Äî but verify that the task-generated policies (which use `Fn::Join` ARN format) are sufficient without the `Fn::GetAtt`-based grant.

## State machine failure semantics

The pipeline chain is purely linear:

```
Write Metadata ‚Üí Synthesize Speech ‚Üí Update Status ‚Üí Done
```

There is no `catch` or `retry` configuration on any of the three task states. If `Synthesize Speech` throws (Polly quota exceeded, invalid parameters, service outage), the execution fails and the DynamoDB record remains `PROCESSING`. The ARCHITECTURE.md describes this table as "a queryable audit trail of all pipeline executions and their outcomes" ‚Äî but the outcome is never recorded on failure.

A minimal fix: add a `Catch` on the Polly task that routes to a parallel `Update Status (FAILED)` DynamoUpdateItem before transitioning to a `Fail` state. This keeps the metadata table accurate regardless of Polly behavior.

## Partition key design and overwrite behavior

The `audioId` field is populated from `$.detail.object.key` ‚Äî the S3 object key of the uploaded file. There is no sort key. This means:

- Re-uploading the same file to S3 triggers a new execution that overwrites the existing DynamoDB record (PutItem replaces the item if the key exists).
- There is no history of previous pipeline runs for the same audio file.
- If the pipeline runs concurrently for the same key (unlikely but possible with rapid re-uploads), the UpdateItem from the first execution could overwrite the PROCESSING status written by the second execution's PutItem, creating a false COMPLETED state.

A composite key like `{audioId, executionId}` using `$$.Execution.Id` would preserve history and prevent cross-execution interference.

## Test coverage gaps

Not tested:
- The ordering of states (Write Metadata before Polly, UpdateStatus after). The definition string tests only check presence, not sequence.
- That the DynamoDB permissions are scoped to *this* table specifically (tests use `Match.arrayWith` on the action without asserting the Resource).
- That `resultPath` values don't collide or clobber input (`$.dynamoResult`, `$.pollyResult`, `$.updateResult` are all distinct, but no test verifies this).
- The actual item schema written by PutItem or the update expression used by UpdateItem.

The snapshot test covers these implicitly as a regression net, but provides no targeted diagnostic when something specific breaks.
