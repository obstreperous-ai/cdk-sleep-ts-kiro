# Step Functions state machine replacing placeholder Lambda with Polly integration

The placeholder Lambda is removed and replaced with a Step Functions state machine that invokes Amazon Polly's `startSpeechSynthesisTask` via the SDK integration pattern (`CallAwsService`). The EventBridge rule now targets the state machine directly, and CloudWatch logging is enabled at level ALL. IAM is scoped to `polly:StartSpeechSynthesisTask` (resource `*`) and `s3:PutObject` on the output bucket.

**Watch for:** The EventBridge event payload shape doesn't match what the Polly task expects â€” the state machine will fail at runtime unless the input is transformed (confirmed). The Polly `iamResources: ['*']` is a wildcard that the ARCHITECTURE.md claims is avoided (confirmed). The `s3:PutObject` grant on the state machine role doesn't help because Polly writes to S3 itself using its own service role, not the caller's credentials (confirmed).

## High-level view

The state machine is a single-task pipeline: EventBridge fires on S3 object creation, starts the state machine, and the sole task calls Polly's async synthesis API. The design is clean for a skeleton â€” one task, one succeed state, CloudWatch logging â€” but the input contract between EventBridge and the Polly task has a gap. The EventBridge event delivers S3 object metadata (`detail.bucket.name`, `detail.object.key`), but the Polly task expects `$.text` at the top level. Without an input transformation on the EventBridge target or a preceding state, every real execution will fail with a JsonPath resolution error.

The IAM posture is mostly sound: CDK's `grantPut` scopes S3 writes to the output bucket ARN, and the Polly action is narrowed to a single API call. However, `grantPut` grants the state machine execution role `s3:PutObject` â€” but Polly's `StartSpeechSynthesisTask` writes to S3 using the Polly service's own credentials, not the caller's. The Polly service needs an `s3:PutObject` permission on its own service role or a bucket policy allowing the Polly service principal. The current `outputBucket.grantPut(stateMachine)` doesn't accomplish what the code intends.

Tests verify resource existence, IAM policy statements, and the EventBridge target wiring. They don't assert the actual state machine definition parameters (bucket name, voice ID, output format) or the input path contract.

## Input contract between EventBridge and the Polly task

The `SfnStateMachine` target passes the full EventBridge event as state machine input. That event has this shape:

```json
{
  "source": "aws.s3",
  "detail-type": "Object Created",
  "detail": {
    "bucket": { "name": "..." },
    "object": { "key": "...", "size": ... }
  }
}
```

The Polly task is configured with `Text: sfn.JsonPath.stringAt('$.text')`, which resolves against the state machine input. There is no `$.text` field in an S3 Object Created event. At runtime, Step Functions will throw a `States.Runtime` error because the JsonPath cannot be resolved.

This is the most significant issue in the change. The skeleton compiles and synthesizes (CDK doesn't validate JsonPath against runtime inputs), but any real execution triggered by an S3 upload will immediately fail. Either the EventBridge target needs an `inputTransformerOptions` that constructs the expected shape, or the state machine needs a preceding Pass state that reshapes the event, or the Polly task parameters should reference fields that actually exist in the event payload.

## Polly's S3 write path vs. the execution role grant

`StartSpeechSynthesisTask` is an async Polly API: you provide an output bucket name, and Polly writes the synthesized audio to that bucket using the Polly service's own credentials. The calling principal (here, the Step Functions execution role) needs `polly:StartSpeechSynthesisTask` permission to invoke the API, but does not perform the S3 write itself.

For Polly to write to the output bucket, the bucket needs a resource policy like:

```json
{
  "Effect": "Allow",
  "Principal": { "Service": "polly.amazonaws.com" },
  "Action": "s3:PutObject",
  "Resource": "arn:aws:s3:::output-bucket/*"
}
```

The current `outputBucket.grantPut(stateMachine)` adds `s3:PutObject` to the state machine's execution role, which is irrelevant to Polly's write path. In production, the synthesis task will succeed (the API call goes through) but the output write will fail with an S3 `AccessDenied` error unless the bucket happens to have a permissive policy.

## IAM and the `iamResources: ['*']` pattern

The `CallAwsService` construct requires an `iamResources` parameter. For Polly, there's no ARN-level resource restriction â€” the API operates on a task ID that doesn't exist until the call completes. So `['*']` is the correct choice here. The ARCHITECTURE.md security section says "avoid wildcard `*` resource permissions where possible; if required, scope them to the narrowest practical actions and resources." The implementation satisfies the second clause (single action), but a brief comment in the code acknowledging why `*` is required would help future readers reconcile the code with the stated policy.

## Test coverage gaps

The five new tests cover: state machine existence, definition string containing "Synthesize Speech", logging configuration, Polly IAM policy statement, and S3 IAM policy statement. The EventBridge target test verifies `Arn` and `RoleArn` are present (any value).

Not tested: the actual parameters inside the state machine definition (output bucket reference, voice ID, output format); that the EventBridge target specifically references the state machine's logical ID (the test accepts any ARN); that the log group is referenced in the logging configuration's destination. A parameter regression â€” say, changing `VoiceId` from `'Joanna'` to something else, or removing `OutputS3BucketName` â€” would not be caught.

The `Match.stringLikeRegexp('.*Synthesize Speech.*Task.*')` assertion is notably loose: any definition string containing those substrings in that order will pass. This is an acknowledged trade-off from the FEAT-002 findings (CDK's `Fn::Join` makes `Match.serializedJson` impractical), but it limits the test's ability to catch regressions in the task configuration.
