# Step Functions state machine with Polly SDK integration (v2)

The placeholder Lambda is replaced with a Step Functions state machine that calls Polly's `startSpeechSynthesisTask` via the CDK `CallAwsService` SDK integration. EventBridge triggers the state machine on S3 object creation, Polly synthesizes a static placeholder text to MP3 in the output bucket, and CloudWatch logging captures all execution data. The four v1 issues — input contract mismatch, missing bucket policy for Polly's write path, undocumented wildcard, and weak test coverage of definition parameters — are all resolved.

**Watch for:** The bucket policy grants `polly.amazonaws.com` unrestricted `PutObject` to the entire output bucket without a condition key scoping writes to objects initiated by this account (likely). The EventBridge target test still accepts any ARN rather than asserting it references the state machine's logical ID (confirmed, minor).

## High-level view

The state machine uses `$.detail.object.key` from the EventBridge event as the Polly output key prefix, meaning each uploaded file triggers synthesis whose output lands alongside the original key namespace. The bucket policy grants `polly.amazonaws.com` the ability to `PutObject` anywhere in the output bucket — functionally correct, but in a shared-account scenario any Polly task (from any principal) could write here without an `aws:SourceAccount` condition.

The IAM model is split between the execution role (`polly:StartSpeechSynthesisTask` with a wildcard, documented as required) and a bucket resource policy for Polly's service-credential write path. The EventBridge target test verifies structure (`Arn` + `RoleArn` present) but not identity — acceptable for a single-machine stack, fragile if the topology grows.

## Bucket policy and cross-account write surface

The bucket policy statement:

```json
{
  "Action": "s3:PutObject",
  "Effect": "Allow",
  "Principal": { "Service": "polly.amazonaws.com" },
  "Resource": "arn:aws:s3:::output-bucket/*"
}
```

When a service principal is trusted without a condition key, any Polly task invocation — potentially from a different account or state machine — can write objects to this bucket. AWS documentation recommends adding `aws:SourceAccount` or `aws:SourceArn` conditions to service-principal trust policies. For a single-account project in early development this is low-risk, but it becomes a real concern if the stack is deployed in a shared account or if the bucket ever holds sensitive output.

## EventBridge target assertion gap

The test asserts:

```typescript
Targets: Match.arrayWith([
  Match.objectLike({
    Arn: Match.anyValue(),
    RoleArn: Match.anyValue(),
  }),
])
```

The presence of `RoleArn` distinguishes this from a Lambda target, but the test does not assert the ARN references `SleepAudioPipelineStateMachine741D1700`. If a future refactor adds a second state machine and accidentally wires the wrong one, this test would pass.
