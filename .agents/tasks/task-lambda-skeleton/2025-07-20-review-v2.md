# Lambda skeleton for audio processing step (v2)

The change inserts a placeholder Lambda function (`SleepAudioProcessor`) into the Step Functions state machine between Polly synthesis and the DynamoDB status update. After v1 review, two fixes were applied: the Process Audio step now has `.addCatch(markFailedTask)` for error routing parity with the Polly step, and the IAM test asserts `Fn::GetAtt` referencing the SleepAudioProcessor ARN rather than accepting any value.

**Watch for:** ARCHITECTURE.md documents error handling only for Polly, not for the newly-added Process Audio catch clause, and the Mermaid diagram omits the ProcessAudio→MarkFailed error edge (confirmed, minor).

**Verdict**: APPROVED

## High-level view

Both blocking issues from v1 are resolved. The synthesized state machine definition confirms Process Audio has a `Catch` block with `ErrorEquals: ["States.ALL"]` routing to Mark Failed, matching the pattern used by Polly. The IAM test now asserts that `lambda:InvokeFunction` targets a resource containing `Fn::GetAtt` with a key matching `SleepAudioProcessor`, which would fail if the resource were broadened to `*`.

The remaining concern is cosmetic: the ARCHITECTURE.md error handling section and the Mermaid diagram haven't been updated to reflect the Process Audio error path. The code is correct; the documentation lags one step behind.

## v1 issue resolution

The synthesized CloudFormation confirms the Process Audio state now includes:

```json
"Catch": [{"ErrorEquals": ["States.ALL"], "ResultPath": "$.errorInfo", "Next": "Mark Failed"}]
```

This means Lambda failures (application errors, timeouts after retry exhaustion, unhandled exceptions) all route into the Mark Failed → Notify Failure → Pipeline Failed path, updating the DynamoDB record and sending an SNS notification. The pipeline can no longer get stuck in PROCESSING indefinitely due to a Lambda failure.

The IAM scoping test asserts:

```typescript
Resource: Match.arrayWith([
  Match.objectLike({
    'Fn::GetAtt': Match.arrayWith([
      Match.stringLikeRegexp('SleepAudioProcessor'),
    ]),
  }),
]),
```

This catches regressions to wildcard (`*`) resources while remaining resilient to CDK logical-ID hash changes.

## Documentation gap in error handling

The ARCHITECTURE.md "Error handling" section reads:

> The Polly task has a Catch clause that routes errors to the Mark Failed state...

It should also mention the Process Audio catch clause. The Mermaid diagram shows `ProcessAudio --> UpdateStatus` (happy path) and `ProcessAudio --> DDB` (data access), but omits `ProcessAudio -->|Error| MarkFailed`. Both are documentation-only fixes with no infrastructure impact.
