# Lambda skeleton for audio processing step

The change inserts a placeholder Lambda function (`SleepAudioProcessor`) into the Step Functions state machine between the Polly synthesis step and the DynamoDB status update. The Lambda receives the full state machine payload via `LambdaInvoke`, logs it, and returns a success envelope. Infrastructure wiring grants the Lambda read/write DynamoDB access and the state machine `lambda:InvokeFunction`. The function does no real work yet — it exists as a seam for future audio validation and metadata enrichment.

**Watch for:** The Lambda has no error-catch integration in the state machine — a failure here leaves the pipeline stuck in PROCESSING with no failure notification (confirmed). The `lambda:InvokeFunction` IAM test asserts only that the permission exists, not that it's scoped to the correct function ARN (confirmed).

## High-level view

The state machine chain grows from five to six steps. `processAudioTask` is wired between Polly and Update Status. CDK's `LambdaInvoke` construct handles the IAM grant automatically, and `grantReadWriteData` gives the Lambda DynamoDB access.

The Process Audio step has no `.addCatch()` clause. Every other fallible step in this pipeline (Polly, SNS success) has explicit error routing. If the Lambda times out or throws, the entire execution fails without updating metadata to FAILED or sending a failure notification. This asymmetry will become a real problem the moment processing logic lands in the handler.

The IAM scoping test for `lambda:InvokeFunction` uses `Match.anyValue()` for the `Resource` field, which means it would pass even if the resource were `*` — defeating the least-privilege verification that other tests in this file provide.

## Missing error catch on the Process Audio step

The state machine definition chains steps as:

```
writeMetadataTask.next(pollySynthesizeTask).next(processAudioTask).next(updateStatusTask)...
```

`pollySynthesizeTask` has `.addCatch(markFailedTask, { resultPath: '$.errorInfo' })`, routing Polly failures into Mark Failed → Notify Failure → Pipeline Failed. `processAudioTask` has no equivalent. If the Lambda throws, times out (default 3s), or returns a runtime error, the Step Functions execution transitions to a terminal `Failed` state with no DynamoDB update and no SNS notification. The metadata record stays stuck in `PROCESSING` forever.

The CDK snapshot shows that `LambdaInvoke` adds default retries for transient Lambda errors (`ClientExecutionTimeoutException`, `ServiceException`, `AWSLambdaException`, `SdkClientException` — 6 retries with exponential backoff). This helps with transient infrastructure failures but doesn't address application-level errors or permanent failures. After retries exhaust, the execution still fails without proper error handling.

## IAM grant pattern

CDK's `LambdaInvoke` construct automatically adds `lambda:InvokeFunction` scoped to the function ARN (plus `:*` for versioned invocations). The snapshot confirms correct scoping:

```json
{
  "Action": "lambda:InvokeFunction",
  "Resource": [
    { "Fn::GetAtt": ["SleepAudioProcessorF6F5D971", "Arn"] },
    { "Fn::Join": ["", [{ "Fn::GetAtt": ["SleepAudioProcessorF6F5D971", "Arn"] }, ":*"]] }
  ]
}
```

The test uses `Resource: Match.anyValue()`, so it would still pass if the resource were broadened to `*`. A tighter assertion like `Match.arrayWith([Match.objectLike({ 'Fn::GetAtt': Match.arrayWith([Match.stringLikeRegexp('SleepAudioProcessor')]) })])` would catch that regression.

## Test coverage gaps

Not tested:

- No assertion that Process Audio appears *between* Synthesize Speech and Update Status in the chain ordering. The test confirms "Process Audio" exists in the definition string but not its position.
- No assertion that `lambda:InvokeFunction` Resource is scoped to the specific function ARN (discussed above).
- No unit test of the Lambda handler. Acceptable for a placeholder, but should be added when real processing logic lands.
