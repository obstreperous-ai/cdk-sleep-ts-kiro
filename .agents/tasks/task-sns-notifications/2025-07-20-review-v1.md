# SNS notification topics for pipeline success and failure

The change adds two KMS-encrypted SNS topics (completed and failed) and wires SNS Publish task states into the Step Functions state machine—one on the success path after the DynamoDB status update, one on the error path after marking the record as failed. The IAM policy is automatically scoped by the CDK L2 construct to allow `sns:Publish` only on the two topic ARNs. Tests cover resource counts, encryption, state machine definition content, and IAM permissions.

**Watch for:** If the "Notify Success" task fails (e.g., SNS throttling, KMS key unavailability), the pipeline execution fails despite having already committed the COMPLETED status to DynamoDB—creating an inconsistency where the metadata says "done" but no notification was emitted and the execution shows as failed (confirmed). The same issue applies to "Notify Failure" but is less consequential there since the execution is already heading toward a Fail state. Hardcoded topic names will collide in multi-stack deployments (confirmed).

## High-level view

The two SNS topics use the AWS-managed `alias/aws/sns` KMS key via `kms.Alias.fromAliasName`, which avoids custom key management but means the stack cannot control key rotation or cross-account access policies. The topics are positioned late in each execution path—after the metadata write—so notification failure doesn't affect data correctness, but it does affect execution outcome on the happy path.

The notification payloads are minimal: pipeline status, audioId, and a timestamp. They omit the error details available in `$.errorInfo` on the failure path, which limits the utility of the failure notification for alerting systems that might want to display root cause without querying the state machine execution history.

Test coverage verifies resource existence and IAM policy presence but doesn't assert the specific topic ARN scoping on the `sns:Publish` statement or the ordering of states in the definition chain.

<details>
<summary>Issues (4)</summary>

1. **Notify Success failure causes false execution failure** — The success notification task has no Catch clause, so an SNS publish error (throttling, KMS issue) fails the entire execution after the DynamoDB record already shows COMPLETED. Add a Catch on `notifySuccessTask` that routes to the Done state, or accept the inconsistency as a known trade-off.
2. **Failure notification omits error context** — `$.errorInfo` (set by the Polly catch clause) is available in the failure path but not included in the SNS message. Subscribers cannot determine failure cause from the notification alone.
3. **Hardcoded topic names** — `topicName: 'SleepAudioPipelineCompleted'` / `'SleepAudioPipelineFailed'` are globally unique per region+account. Multi-stack deployments in the same account/region will fail. Consider omitting `topicName` to let CloudFormation generate unique names, or incorporate the stack name.
4. **sns:Publish test doesn't assert resource scoping** — The test matches `Action: 'sns:Publish', Effect: 'Allow'` but doesn't verify the `Resource` field is scoped to the topic ARNs. A future refactor that accidentally grants `Resource: '*'` would still pass.

## Notification failure on the success path

The state machine definition chains `updateStatusTask → notifySuccessTask → Done` with no error handling on `notifySuccessTask`:

```typescript
writeMetadataTask.next(pollySynthesizeTask).next(updateStatusTask).next(notifySuccessTask).next(new sfn.Succeed(this, 'Done'))
```

If `notifySuccessTask` throws (SNS throttle, transient KMS error, service outage), Step Functions marks the entire execution as failed. At that point, the DynamoDB record already reads `status=COMPLETED`. An external observer checking DynamoDB sees success; the execution history shows failure. Retry logic at the EventBridge layer would re-trigger the entire pipeline for an already-processed file.

The failure path has the same structural issue (`markFailedTask → notifyFailureTask → Fail`), but since the execution is heading toward a Fail state regardless, the practical impact is smaller—the only loss is that subscribers don't receive the notification, and the execution fails at a different state than expected.

Adding a Catch clause on `notifySuccessTask` that transitions to Done would make notification best-effort without affecting execution outcome. This is a common pattern for non-critical side-effect tasks in Step Functions.

## Notification payload completeness

The success message includes `{ pipelineStatus, audioId, completedAt }` and the failure message includes `{ pipelineStatus, audioId, failedAt }`. On the failure path, the state machine has `$.errorInfo` populated by the Polly catch clause (containing the error name and cause), but this context is not forwarded to the SNS message.

For an alerting system subscribed to the failed topic, the notification tells it *that* something failed but not *what* failed. The subscriber must call `DescribeExecution` or query the logs to determine root cause. Including `error.$: '$.errorInfo.Error'` and `cause.$: '$.errorInfo.Cause'` in the failure message would make the notification self-contained for alerting.

## Hardcoded topic names and multi-environment deployments

Both topics specify explicit `topicName` values:

```typescript
const completedTopic = new sns.Topic(this, 'PipelineCompletedTopic', {
  topicName: 'SleepAudioPipelineCompleted',
  ...
});
```

SNS topic names must be unique within an account+region. If this stack is deployed multiple times (staging and production in the same account, or blue/green deployments), the second deployment will fail with `TopicAlreadyExists`. Omitting `topicName` lets CloudFormation generate a unique physical name from the logical ID and stack name, which is the CDK convention for resources that don't need stable physical names for external integrations.

## Test coverage gaps

The SNS-related tests verify:
- Resource count (2 topics)
- KMS encryption presence (any value for `KmsMasterKeyId`)
- State machine definition contains "Notify Success" and "Notify Failure" strings
- IAM policy contains an `sns:Publish` / `Allow` statement

What's not tested:
- The `sns:Publish` resource scoping—the test doesn't assert that the Resource field references the specific topic ARNs rather than a wildcard. The synthesized template does scope correctly (`"Resource": [{"Ref": "PipelineCompletedTopicA74D8D79"}, {"Ref": "PipelineFailedTopic64B61AAF"}]`), but the test wouldn't catch a regression that broadens the scope.
- The ordering of states in the chain (Notify Success comes after Update Status, not before it).
- The content of the SNS message payload (audioId field, timestamp field, pipelineStatus value).

The two "KMS encryption" tests are effectively identical in what they prove—the first asserts *at least one* topic has `KmsMasterKeyId`, the second asserts both do via `findResources`. The first test is redundant with the second.
