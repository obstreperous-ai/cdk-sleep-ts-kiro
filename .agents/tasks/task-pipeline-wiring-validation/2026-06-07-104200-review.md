# Two-layer input validation with pipeline reorder

The pipeline now rejects bad inputs early through a Step Functions Choice state ("Validate Input") that checks for bucket name, object key, and supported audio extensions before any compute runs. The Lambda was also hardened: it throws on invalid inputs instead of returning a 500-level response, which lets the Step Functions Catch clause route failures to the Mark Failed path consistently. The pipeline order changed from WriteMetadata -> Polly -> Lambda to WriteMetadata -> Validate -> Lambda -> Polly, placing validation and processing before the speech synthesis step.

**Watch for:** The Notify Failure task references `$.errorInfo.Error` and `$.errorInfo.Cause`, but the Choice-state "otherwise" path reaches Mark Failed without setting `$.errorInfo` — the failure notification will fail or produce null fields (confirmed). The Lambda extension check uses `lastIndexOf('.')` which produces the entire filename for extensionless files, yielding a confusing error message (possible). The `INPUT_BUCKET_NAME` and `OUTPUT_BUCKET_NAME` env vars are wired but never consumed for any logic (confirmed, minor).

## High-level view

The Choice state provides a zero-compute gate: if the EventBridge payload lacks a bucket name, object key, or has an unsupported extension, the execution routes directly to Mark Failed without invoking Lambda or Polly.

The Lambda's error-handling posture changed from returning a 500 response object (which Step Functions treats as success) to throwing, which correctly triggers the Catch clause and routes to Mark Failed.

The Notify Failure path has a structural gap: executions that fail via the Choice "otherwise" branch arrive at Mark Failed without `$.errorInfo` in the state, but Notify Failure references `$.errorInfo.Error` and `$.errorInfo.Cause`. Step Functions will throw a `States.Runtime` error when resolving missing JSONPath, causing the notification itself to fail.

The Choice state's extension matching is case-sensitive (`StringMatches` uses literal glob), while the Lambda normalizes to lowercase. A file like `audio.WAV` gets rejected at the Choice layer with a generic failure message rather than reaching the Lambda's descriptive error. Not a bug, but an observability asymmetry.

## The `$.errorInfo` gap on the Choice-otherwise path

The Notify Failure task constructs its SNS message using:

```typescript
error: sfn.JsonPath.stringAt('$.errorInfo.Error'),
cause: sfn.JsonPath.stringAt('$.errorInfo.Cause'),
```

When a Lambda or Polly task fails and its Catch fires, the error details are placed at `$.errorInfo` via `resultPath: '$.errorInfo'`. But the Choice state's `.otherwise(markFailedTask)` path is a direct transition — there's no error object, no catch, no resultPath injection. The execution arrives at Mark Failed with the original state (just `$.detail` and `$.dynamoResult`), then proceeds to Notify Failure which tries to read `$.errorInfo.Error`.

Step Functions will throw a `States.Runtime` error when it can't resolve a JSONPath, which would cause the Notify Failure task itself to fail. Since Notify Failure has no Catch, the entire execution fails with an opaque runtime error rather than a clean Pipeline Failed terminal state. This means validation rejections — the most common failure mode — produce the worst observability outcome.

The fix is either: (a) insert a Pass state on the Choice-otherwise path that sets `$.errorInfo` to `{"Error": "ValidationError", "Cause": "Input failed validation checks"}`, or (b) restructure Notify Failure to use optional JSONPath references or a static message for the validation case.

## Extension matching asymmetry between Choice and Lambda

The Choice state uses `StringMatches('*.wav')` — Step Functions `StringMatches` is case-sensitive, so `file.WAV` would fail the Choice validation and route to Mark Failed.

The Lambda uses `.toLowerCase()` before comparing against the allowed list, so it's case-insensitive. A file like `audio.WAV` would:
1. Fail the Choice state (case-sensitive glob doesn't match `*.wav`)
2. Route to Mark Failed
3. Never reach the Lambda's descriptive error message

The Lambda's case-insensitivity is unreachable — the stricter Choice state gates it. The file still gets rejected, but via the Choice path which (compounding with issue #1) produces a runtime error at the Notify Failure step rather than a clean failure message. If case-insensitive matching is ever desired, the Choice conditions would need variants for each case, which is impractical — relaxing the Choice to only check for a dot-something suffix and letting the Lambda handle extension validation would be more maintainable.

## Test coverage gaps

Not tested: the runtime behavior of the Lambda validation logic (no unit tests for the handler itself), the `$.errorInfo` gap on the Choice-otherwise path, and case-sensitivity behavior of the extension matching.
