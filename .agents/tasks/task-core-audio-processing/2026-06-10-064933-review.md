# Full audio processing pipeline in Lambda

The `SleepAudioProcessor` Lambda was upgraded from a validation-only skeleton to a complete processing pipeline: download from S3, branch on file type (Polly synthesis for text, passthrough for audio), upload to the output bucket, and write completion metadata to DynamoDB. The CDK stack was updated to grant the necessary S3 and Polly permissions and bump memory/timeout to accommodate real I/O workloads. A new 518-line test file mocks the AWS SDK at the client boundary and exercises both paths plus error propagation.

Watch for: The Lambda downloads from whichever bucket the event names (`event.detail.bucket.name`) but IAM only grants read on the CDK-provisioned input bucket — any cross-account or alternate-bucket trigger will fail-closed at IAM (confirmed). Polly `SynthesizeSpeech` has a hard 3 000-character input limit; no guard exists, so large `.txt` files will produce an opaque Polly error rather than a clear validation message (confirmed). The `audioId` used as the DynamoDB key is the raw S3 object key, which can contain characters that complicate downstream queries (likely).

## High-level view

The Lambda now runs a four-step pipeline — S3 get, Polly-or-passthrough, S3 put, DynamoDB update — within a single synchronous invocation. The design loads the entire input file into memory; the 512 MB allocation caps practical file size at roughly 250–300 MB after runtime overhead.

IAM is scoped to the CDK-stack-managed buckets via `grantRead`/`grantWrite` plus a wildcard Polly policy. The event-supplied bucket name is not validated against `INPUT_BUCKET_NAME`, so the IAM layer is the only enforcement, producing confusing AccessDenied errors rather than a descriptive validation failure if there's a mismatch.

Polly's `SynthesizeSpeech` (synchronous) imposes a 3 000-character ceiling. There is no pre-flight check, so oversized text will trip a Polly service error, get caught by the generic handler, and surface as a Step Functions failure with no indication that the input was simply too long.

DynamoDB metadata is updated only on success. If the upload succeeds but the metadata write fails, S3 contains an orphan object with no tracking record, and Step Functions will retry the entire Lambda (re-uploading under a new timestamped key).

## Polly text-length ceiling

The Lambda calls `SynthesizeSpeech` (the synchronous API) rather than `StartSpeechSynthesisTask` (the async API that supports up to 200 000 characters via S3). The synchronous endpoint has a hard 3 000-character maximum. The handler applies no length check before invoking Polly:

```typescript
const textContent = inputBody.toString('utf-8');

const pollyResponse = await pollyClient.send(new SynthesizeSpeechCommand({
  Engine: 'neural',
  VoiceId: 'Joanna',
  OutputFormat: 'mp3',
  Text: textContent,
}));
```

A text file with 3 001 characters will hit a Polly `TextLengthExceededException`. The generic catch block re-throws it, Step Functions retries it three times (per the retry policy), and all three retries fail identically — wasted compute and confusing observability noise. Either validate length and fail fast with a descriptive message, or use `StartSpeechSynthesisTask` for the text path (though that changes the execution model to async polling). **Confidence: confirmed** — the limit is documented AWS behavior and the code has no guard.

## Event bucket vs. IAM scope mismatch

The handler reads `event.detail.bucket.name` and issues `GetObjectCommand` against it. The IAM grant is scoped to the CDK-provisioned input bucket only:

```typescript
inputBucket.grantRead(audioProcessorFunction);
```

If the Step Functions state machine is invoked manually or via a different trigger with an arbitrary bucket name, the request will fail at IAM with AccessDenied rather than a clear application-level error. A one-line guard (`if (bucketName !== inputBucketName) throw ...`) would make debugging simpler. **Confidence: confirmed** — observable from the code and CDK wiring.

## Non-atomic upload-then-metadata sequence

The processing pipeline runs: S3 download → transform → S3 upload → DynamoDB update. If the DynamoDB write fails (throttle, network hiccup), the S3 object already exists in the output bucket. The Step Functions retry will re-execute the entire Lambda, generating a new timestamp and uploading a second copy. Over retries, this could produce up to four orphan objects (three retries + original) with no metadata pointing to them.

A deterministic output key (e.g., based on `audioId` alone without a timestamp) would make the operation idempotent — re-uploads would overwrite the same key. The trade-off is losing the audit trail of multiple processing attempts, but that information could live in DynamoDB or logs instead. **Confidence: confirmed** — the `Date.now()` timestamp guarantees a unique key per invocation.

## Memory constraints for audio passthrough

The handler downloads the entire file into memory (`streamToBuffer`) then uploads it. No ceiling is enforced: a malformed event pointing to a 500 MB file would OOM the Lambda. The `GetObjectCommand` response includes `ContentLength` — a pre-flight check like `if (getObjectResponse.ContentLength > MAX_SIZE) throw ...` would fail fast with a meaningful error rather than a Lambda timeout or OOM crash. **Confidence: likely** — depends on whether upstream validation constrains file size; from the diff it doesn't appear to.

## Test coverage gaps

Not tested: behavior when `process.env.TABLE_NAME` or `process.env.OUTPUT_BUCKET_NAME` is undefined (the handler would pass `undefined` to `PutObjectCommand.Bucket` and `UpdateCommand.TableName`, producing confusing SDK errors). Not tested: Polly returning a `null` or empty `AudioStream`. Not tested: S3 returning `Body` as `undefined` (possible for zero-byte objects). These edge cases would produce unhandled exceptions rather than descriptive errors.
