# Full audio processing pipeline in Lambda (v2)

The `SleepAudioProcessor` Lambda implements a four-step pipeline — S3 download, Polly synthesis or audio passthrough, S3 upload, DynamoDB metadata update — with validation guards for file size (100 MB cap), Polly text length (3 000 chars), and event bucket matching. The CDK stack grants least-privilege S3 and Polly permissions and sizes the Lambda at 512 MB / 120 s. A 619-line test file mocks all AWS SDK clients and exercises both processing paths, boundary conditions, and error propagation.

Watch for: The `streamToBuffer` helper uses `any` for the stream parameter with no null check; if Polly returns a `null` `AudioStream` or S3 returns `undefined` `Body`, the `for await` will throw an unhandled `TypeError` rather than a descriptive processing error (likely). The file-size guard only fires when `ContentLength` is defined — a missing header bypasses protection entirely (possible).

## High-level view

Two residual gaps remain around stream handling. The `streamToBuffer` utility assumes its argument is always a valid async iterable — no defensive check for `null` or `undefined`. For the S3 download path this will surface as a confusing `TypeError` if S3 returns an empty `Body`; for the Polly path it's the same story with `AudioStream`. Separately, the `ContentLength` check only fires when the header is defined, which is the common case but not guaranteed for all S3 object configurations.

The `outputKey` metadata stored in DynamoDB is an S3 URI (`s3://bucket/key`) while the Lambda's return value `outputKey` is a plain key path. This asymmetry could confuse downstream consumers depending on which field they read.

## v1 fix verification

Issue 1 (Polly 3 000-char limit): Fixed. `MAX_POLLY_TEXT_LENGTH = 3000` with a pre-flight `textContent.length` check. Tested at boundary (3000 passes, 3001 fails).

Issue 2 (Event bucket validation): Fixed. Compares `bucketName !== inputBucketName` before any S3 call. Tested with a wrong-bucket event.

Issue 4 (File size guard): Fixed. `MAX_FILE_SIZE = 100 * 1024 * 1024` with an upfront `ContentLength` check. Tested at boundary (100 MB passes, 150 MB fails).

Issues 3 and 5: Accepted as design decisions — not re-raised.

## Stream null safety in `streamToBuffer`

The `streamToBuffer` helper is typed `(stream: any): Promise<Buffer>` and immediately enters a `for await` loop:

```typescript
async function streamToBuffer(stream: any): Promise<Buffer> {
  const chunks: Uint8Array[] = [];
  for await (const chunk of stream) {
    chunks.push(typeof chunk === 'string' ? Buffer.from(chunk) : chunk);
  }
  return Buffer.concat(chunks);
}
```

Two call sites pass potentially nullable values: `getObjectResponse.Body` (typed `Readable | ReadableStream | Blob | undefined` in the SDK) and `pollyResponse.AudioStream` (typed `Readable | undefined`). If either is `undefined`, the `for await` throws `TypeError: undefined is not async iterable`. This surfaces as a generic error in the catch block rather than a specific "Polly returned no audio" or "S3 object has no body" message. The test suite doesn't cover this edge case. **Confidence: likely** — the SDK types allow `undefined` and the code has no guard.

## ContentLength gap in the file-size check

```typescript
const contentLength = getObjectResponse.ContentLength;
if (contentLength !== undefined && contentLength > MAX_FILE_SIZE) {
  throw new Error(...);
}
```

The `!== undefined` condition means objects without a `ContentLength` header bypass the check entirely. In that scenario, the Lambda will buffer the entire object regardless of size — the OOM scenario the guard was meant to prevent. A fallback check on `inputBody.length` after buffering (or treating undefined as a failure) would close the gap. **Confidence: possible** — standard S3 PutObject always sets ContentLength; the gap only manifests with unusual upload patterns or non-standard S3 endpoints.

## DynamoDB outputKey vs Lambda response outputKey

The DynamoDB update stores:
```typescript
':outputKey': `s3://${outputBucketName}/${outputKey}`,
```

While the Lambda response returns:
```typescript
outputKey,  // plain key like "processed/audio/sleep-sounds-1718012345.mp3"
```

A downstream service reading the DynamoDB record gets `s3://bucket/processed/...` but the Step Functions state machine (which receives the Lambda response) gets just `processed/...`. Any consumer relying on both sources expecting the same format will need to handle both conventions. **Confidence: confirmed** — observable in the code.
