# Error handling, retry policies, and observability for the sleep audio pipeline

The change adds four behavioral capabilities to the CDK Step Functions pipeline: exponential-backoff retries on all four task states, specific-error Catch blocks on Write Metadata and Update Status (previously only Process Audio and Polly had them), X-Ray tracing on both the Lambda and the state machine, and CloudWatch alarms for execution failures and Lambda errors. The Lambda handler switches from unstructured `console.log` to structured JSON logging with request correlation.

**Watch for:** The Process Audio task now carries two overlapping retry policies (CDK's built-in LambdaInvoke default plus the custom one), resulting in up to 9 total retries for `Lambda.ServiceException` ‚Äî likely unintentional (confirmed). The Polly and Process Audio Catch blocks were narrowed from catch-all (`States.ALL`) to specific error types, meaning errors outside those types now propagate as unhandled execution failures instead of routing to Mark Failed (confirmed).

## High-level view

The retry strategy is straightforward for the DynamoDB tasks ‚Äî `States.ALL` with 3 attempts and exponential backoff. For the Lambda and Polly tasks, the intent is to retry on transient/service errors but the implementation creates coverage gaps. The CDK `LambdaInvoke` construct injects its own default retry policy (6 attempts for four Lambda-specific error classes), and the custom `addRetry` stacks on top of it rather than replacing it, leading to overlapping error matchers and inflated retry counts.

The Catch narrowing is the more consequential change. Previously both `processAudioTask` and `pollySynthesizeTask` caught all errors. Now they only catch explicitly listed types. Polly could surface throttling errors (`Polly.ThrottlingException`) or access denied that won't match `States.TaskFailed` ‚Äî those would bypass the Mark Failed path entirely, leaving the DynamoDB record stuck in PROCESSING while the execution reports failure.

Both CloudWatch alarms lack `alarmActions` ‚Äî they transition to ALARM state but don't notify anyone. The existing `failedTopic` SNS topic is a natural target.

## Retry policy overlap on Process Audio

The CDK `LambdaInvoke` construct automatically adds a retry policy targeting `Lambda.ClientExecutionTimeoutException`, `Lambda.ServiceException`, `Lambda.AWSLambdaException`, and `Lambda.SdkClientException` with `MaxAttempts: 6` and `BackoffRate: 2`. The synthesized state machine definition confirms this:

```
"Retry":[
  {"ErrorEquals":["Lambda.ClientExecutionTimeoutException","Lambda.ServiceException",
   "Lambda.AWSLambdaException","Lambda.SdkClientException"],
   "IntervalSeconds":2,"MaxAttempts":6,"BackoffRate":2},
  {"ErrorEquals":["States.TaskFailed","Lambda.ServiceException","Lambda.SdkClientException"],
   "IntervalSeconds":2,"MaxAttempts":3,"BackoffRate":2}
]
```

Step Functions evaluates retry entries in array order. A `Lambda.ServiceException` matches the first entry and retries 6 times. If it still fails, it then matches the second entry and retries 3 more times ‚Äî 9 total attempts with exponential backoff starting from 2 seconds. The ARCHITECTURE.md documents "MaxAttempts 3" which doesn't match the actual behavior.

The fix is to pass `retryOnServiceExceptions: false` to the `LambdaInvoke` constructor and let the custom retry be the sole retry policy.

## Catch scope narrowing

Before this change, both `processAudioTask.addCatch` and `pollySynthesizeTask.addCatch` omitted the `errors` parameter, producing `"ErrorEquals": ["States.ALL"]` in the synthesized definition ‚Äî a catch-all. Now they specify explicit lists:

- Process Audio catches: `States.TaskFailed`, `Lambda.ServiceException`, `Lambda.SdkClientException`
- Polly catches: `States.TaskFailed`

This means:
- For Process Audio: `Lambda.ClientExecutionTimeoutException` and `Lambda.AWSLambdaException` are retried (by the default policy) but never caught. After exhausting retries, these become unhandled failures.
- For Polly: any SDK error that isn't `States.TaskFailed` (throttling, access denied, internal service errors surfaced with their original SDK exception class) bypasses Mark Failed.

The consequence is that the DynamoDB metadata record stays in PROCESSING indefinitely for uncaught error types, while the execution itself reports failure. The alarm fires, but the data is inconsistent.

## Alarms without notification targets

Both CloudWatch alarms are created without `alarmActions`, `okActions`, or `insufficientDataActions`. They'll transition to ALARM state but won't notify anyone. The stack already has `failedTopic` (an SNS topic with KMS encryption) that could serve as the alarm action target.

## Test fragility on definition serialization

The retry/catch tests parse the synthesized state machine definition by joining the `Fn::Join` array parts and using regex to extract configurations from specific task sections. The regex patterns like `/"Process Audio":\{[^]*?"Retry":\[([^]*?)\],"Catch"/` depend on the exact field order in the serialized definition. If CDK changes serialization order (e.g., `Catch` before `Retry`), these tests break silently ‚Äî they'd fail to match rather than catching a real regression.
