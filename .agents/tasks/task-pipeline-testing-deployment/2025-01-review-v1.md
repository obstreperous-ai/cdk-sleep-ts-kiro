# Multi-environment support, pipeline skeleton, and expanded test coverage

This change introduces multi-environment deployment via CDK context (`--context environment=<env>`), adds an optional CDK Pipelines stack gated by `--context enablePipeline=true`, tunes Lambda memory/timeout and state machine timeout/name, and adds 19 new tests covering the end-to-end pipeline flow, input validation paths, error routing, IAM least-privilege, and multi-environment tagging. The CI workflow is updated to pass `--context environment=dev` to synth and diff.

**Watch for:** Duplicate environment tagging logic applied in two places that can silently diverge (confirmed). The PipelineStack has minimal test coverage — only resource count, no assertion on source config or deployment stage (confirmed). The `stateMachineName` is hardcoded, which will cause CloudFormation conflicts if two environments deploy to the same account/region (likely).

## High-level view

Environment context is read via `tryGetContext('environment')` and defaults to `dev`. The value is applied as a tag on all resources. However, tagging happens both in `bin/cdk-base.ts` (on the stack externally) and inside `CdkBaseStack`'s constructor (on `this`), creating a redundancy that will diverge if someone changes the logic in only one place.

The PipelineStack is a thin CDK Pipelines wrapper: CodePipeline sourcing from a GitHub connection, a ShellStep synth, and one deployment stage containing `CdkBaseStack`. It's gated behind a context flag so it doesn't interfere with normal deployments. Its test file asserts only that a pipeline resource exists; the stage wiring and synth commands are untested.

Lambda memory (256 MB) and timeout (30 s) are now explicit, and the state machine gets a 15-minute global timeout and a fixed name (`SleepAudioPipeline`). The fixed name creates a uniqueness constraint — two stacks in the same account/region will collide.

The test suite gains coverage for the full happy-path chain ordering, input validation per extension, error path routing, EventBridge rule scoping, IAM wildcard auditing, and environment tag propagation. The IAM test programmatically verifies that wildcard resources are limited to documented exceptions — a pattern that catches future permission creep automatically.

## Duplicate environment tagging in bin vs stack

The environment tag is applied twice: once in `bin/cdk-base.ts` at the app level (`cdk.Tags.of(stack).add(...)`) and once inside `CdkBaseStack`'s constructor (`cdk.Tags.of(this).add(...)`). CDK deduplicates identical tag values, so there's no functional bug today. The problem is maintenance: the two sites read context independently and default independently.

The `SleepAudioStage` in `PipelineStack` instantiates `CdkBaseStack` without any external tagging — it relies entirely on the stack-internal tagging. The bin-level tag only covers the standalone deployment path. This asymmetry confirms the bin-level tag is redundant.

```
// bin/cdk-base.ts — redundant because CdkBaseStack applies it internally
const stack = new CdkBaseStack(app, 'CdkBaseStack');
cdk.Tags.of(stack).add('environment', environment);  // ← remove this

// lib/cdk-base-stack.ts — authoritative tag site
const environment = this.node.tryGetContext('environment') || 'dev';
cdk.Tags.of(this).add('environment', environment);
```

## StateMachineName uniqueness constraint

The state machine now has `stateMachineName: 'SleepAudioPipeline'`. CloudFormation requires state machine names to be unique within an account+region. If the same stack is deployed to the same account for multiple environments (a common pattern with CDK context-based env switching), the second deployment will fail with an `AlreadyExists` error.

The fix is either to incorporate the environment into the name (`SleepAudioPipeline-${environment}`) or to drop the hardcoded name entirely and let CFN auto-generate one. The latter is safer for multi-stack scenarios and avoids rename failures during refactoring.

## Pipeline synth without environment context

The `ShellStep` in `PipelineStack` runs:
```typescript
commands: ['npm ci', 'npx cdk synth'],
```

The pipeline-synthesized stack will always use the default `dev` environment context value. If the pipeline is extended to deploy to staging or prod, the synth command needs `--context environment=<target>` or per-stage context injection.

## PipelineStack test coverage

`test/pipeline-stack.test.ts` asserts only:
```typescript
template.resourceCountIs('AWS::CodePipeline::Pipeline', 1);
```

This verifies the stack synthesizes but doesn't catch regressions in source configuration, synth commands, or stage wiring. A minimal improvement would assert that the pipeline has at least one deployment stage.
