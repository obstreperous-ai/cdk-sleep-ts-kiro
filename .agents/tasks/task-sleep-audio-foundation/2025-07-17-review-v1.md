# Sleep audio pipeline foundation: S3 buckets, EventBridge rule, placeholder Lambda

This change lays the groundwork for an event-driven audio processing pipeline. Two S3 buckets (input and output) are provisioned with S3-managed encryption, versioning, and full public-access blocking. The input bucket emits EventBridge notifications on object creation, which an EventBridge rule routes to a placeholder Lambda. The stack is tested via CDK assertions (property-level and snapshot), and ARCHITECTURE.md is updated to distinguish "currently implemented" from "planned" resources.

**Watch for:** Tests don't distinguish between the two buckets, so property regressions on one bucket could be masked by the other still matching (confirmed). No lifecycle or removal policies are set on the buckets, so `cdk destroy` will fail if the buckets contain objects (confirmed). The EventBridge rule has no object-key filter, meaning all uploads (including non-audio files) trigger the target (confirmed).

## High-level view

The EventBridge integration relies on CDK's `eventBridgeEnabled` bucket property, which deploys a custom resource (a CDK-managed Python Lambda) to configure S3 bucket notifications. The stack therefore contains two Lambda functions, not one — something operators should be aware of when reading CloudFormation output or debugging deploy failures.

The test suite verifies resource properties using partial-match assertions (`hasResourceProperties`). Because both buckets share identical encryption and versioning configuration, most "Input Bucket" tests would also pass if only the output bucket existed. The tests confirm that *at least one* bucket has the expected properties, not that each specific bucket does.

3. **EventBridge rule matches all Object Created events** — The rule has no prefix or suffix filter, so any object (including temporary multipart fragments or non-audio files) will invoke the target. Not a bug in a stub phase, but the rule will need `detail.object.key` filtering before the Lambda does real work. (confirmed)
4. **Placeholder Lambda has no timeout or memory constraints** — Defaults to 3 seconds timeout and 128 MB, which is fine for a no-op but should be explicitly set before the function gains real logic, to avoid silent Lambda timeouts in production. (confirmed)

## Test ambiguity between input and output buckets

The test file organizes assertions into `describe('S3 Input Bucket')` and `describe('S3 Output Bucket')` blocks, but the assertions themselves use `template.hasResourceProperties('AWS::S3::Bucket', {...})` without any identifying property. Both buckets have identical encryption, versioning, and public-access settings. If the output bucket's encryption were accidentally removed, the "Output Bucket" test would still pass because the input bucket matches. The snapshot test catches this indirectly, but property-level tests should provide independent safety.

One remedy: match against the logical ID prefix using `template.hasResource('AWS::S3::Bucket', { Properties: {...} })` with a `Match` filter, or assert `template.resourceCountIs('AWS::S3::Bucket', 2)` alongside the property checks.

## Bucket retention and lifecycle

The synthesized template shows `"DeletionPolicy": "Retain"` and `"UpdateReplacePolicy": "Retain"` on both buckets. This is correct for production stateful resources, but means `cdk destroy` in development will leave behind buckets that must be manually deleted. With versioning enabled and no lifecycle rules, object versions accumulate without bound — a cost concern for a pipeline that will ingest audio files continuously.

## EventBridge custom resource and snapshot noise

Enabling `eventBridgeEnabled: true` causes CDK to synthesize a `Custom::S3BucketNotifications` resource backed by a dedicated Python Lambda that calls `s3:PutBucketNotification`. The snapshot captures the entire inline Python handler source (~80 lines), which means snapshot diffs will be noisy whenever CDK upgrades change that handler code. This is standard CDK behavior, but teams relying on snapshot stability should be aware that this resource is outside their direct control.
